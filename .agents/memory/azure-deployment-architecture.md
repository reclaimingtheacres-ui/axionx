## Production deployment ‚Äî Azure App Service

- App Service name: `axionx-prod`
- Live domain: `https://axionx.com.au`
- Startup: `startup.sh` ‚Üí `gunicorn wsgi:application` on port 8000
- Persistent DB: `/home/site/data/axion.db` (mounted storage, NOT ephemeral)
- Production builds use Oryx (`SCM_DO_BUILD_DURING_DEPLOYMENT=true`)

**Why:** The Replit dev server (`.replit.dev`) is only the test/preview environment. All real users are on Azure. "Deploying" in Replit does NOT update Azure.

## GitHub Actions ‚Äî authoritative workflow

- **Active:** `.github/workflows/main_axionx-prod.yml`
  - Auto-generated by Azure portal on 2026-03-02
  - Secret: `AZUREAPPSERVICE_PUBLISHPROFILE_3B6583AC3C6343A2944A51F74544EA8C`
  - Triggers: `push` to `main` + `workflow_dispatch`
  - Two jobs: build ‚Üí deploy (excludes `antenv/`, targets `slot-name: Production`)
- **Disabled:** `.github/workflows/azure-deploy.yml`
  - `if: false` added to job on 2026-06-11 (commit 735bdfc)
  - Was a manual precursor created 2026-03-01, superseded the next day by the portal workflow
  - Had a different secret (`AZURE_WEBAPP_PUBLISH_PROFILE`) causing a race condition on every push

**Why:** Having both active caused a race condition ‚Äî two workflows deploying to the same App Service slot on every push to main, using different publish profiles.

## Critical: download-artifact@v4 path convention

`upload-artifact@v4` with `name: python-app` + `path: .` uploads all workspace files flat.
`download-artifact@v4` with NO `path:` key extracts into `$GITHUB_WORKSPACE/python-app/` (adds a named subdirectory automatically).
`webapps-deploy@v3` with `package: .` then deploys the parent ‚Äî sending `python-app/app.py` NESTED one level deep. Azure never updates its root `app.py`.

**The fix (both keys are required):**
```yaml
- uses: actions/download-artifact@v4
  with:
    name: python-app
    path: python-app        # ‚Üê force extract INTO named subdirectory

- uses: azure/webapps-deploy@v3
  with:
    package: python-app     # ‚Üê deploy the subdirectory's CONTENTS, not parent
```

**Why:** Without `path: python-app` on download, v4 creates a `python-app/` subdirectory anyway. Without `package: python-app` on deploy, the entire workspace (including the subdirectory) is zipped, nesting `app.py` one level deep. Azure Oryx reuses its cached build at `/tmp/XXXX/` and the new code never runs.

## OneDeploy restart behaviour

OneDeploy marking "success" in GitHub Actions logs does NOT guarantee an immediate app restart. Sometimes a second `workflow_dispatch` run is required to unstick the App Service and load the new code. Symptom: all GH Actions steps green, but the new route still returns 404 (Flask 404 from AxionX template, confirming the running code is old). Fix: trigger a second `workflow_dispatch` run.

## Replit deployment (separate, NOT production)

- URL: `https://AaxionConsole.replit.app` ‚Äî a Replit-managed autoscale deployment
- Has never had a successful build (image > 8 GiB until `.replignore` was added)
- `.replignore` added in commit `eed8b70` to exclude `.pythonlibs` (~7.5 GiB) and `.cache` (~4.1 GiB)
- This is NOT where real users go ‚Äî it is a parallel Replit-only deployment

## How to deploy to production (Azure)

1. Push commits to `origin/main` ‚Äî GitHub Actions `main_axionx-prod.yml` fires automatically
2. OR go to GitHub ‚Üí Actions ‚Üí "Build and deploy Python app to Azure Web App - axionx-prod" ‚Üí Run workflow ‚Üí branch: `main`

**How to apply:**
- Never assume "deploying in Replit" = deploying to the live site
- After pushing any feature, verify on Azure directly (`axionx.com.au`) ‚Äî unauthenticated hit on a protected route should return 302, not 404
- If Azure returns 404 (Flask template 404 served by gunicorn) after a "successful" deploy, trigger a second `workflow_dispatch` run to force a proper restart
- Check GitHub Actions for `main_axionx-prod.yml` run status; `azure-deploy.yml` must always be skipped
